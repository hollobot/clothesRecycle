## 1. 角色与权限

### 1.1 角色定义

| 角色       | 标识           | 说明                                 |
| ---------- | -------------- | ------------------------------------ |
| 超级管理员 | `SUPER_ADMIN`  | 全系统配置，管理多校区、管理员账号   |
| 校区管理员 | `CAMPUS_ADMIN` | 管理本校区回收点、物品审核、用户管理 |
| 普通学生   | `USER`         | 发布、浏览、认领衣物，管理个人积分   |

### 1.2 权限矩阵

| 功能               | 超级管理员 | 校区管理员  | 普通学生 |
| ------------------ | ---------- | ----------- | -------- |
| 校区管理（增删改） | ✅          | ❌           | ❌        |
| 管理员账号管理     | ✅          | ❌           | ❌        |
| 全平台数据看板     | ✅          | ❌           | ❌        |
| 回收点管理         | ✅          | ✅（本校区） | ❌        |
| 物品审核           | ✅          | ✅（本校区） | ❌        |
| 用户管理           | ✅          | ✅（本校区） | ❌        |
| 本校区数据看板     | ✅          | ✅           | ❌        |
| 发布物品           | ❌          | ❌           | ✅        |
| 浏览/认领物品      | ❌          | ❌           | ✅        |
| 积分兑换/明细      | ❌          | ❌           | ✅        |

### 1.3 Sa-Token 鉴权方案

- 登录后颁发 Token，存入 Redis（TTL 7 天，滑动续期）
- 使用 `@SaCheckRole` 注解做接口级权限控制
- 管理后台与用户端使用独立的 Sa-Token 上下文（`StpUtil` / 自定义 `StpLogic`）

## 3. 用户端（Web，适配手机）

### 3.1 账号模块

#### 3.1.1 注册

**页面：** `/register`

| 字段       | 类型   | 规则                        |
| ---------- | ------ | --------------------------- |
| 手机号     | input  | 必填，格式校验，唯一        |
| 短信验证码 | input  | 必填，60s 有效，Redis 存储  |
| 密码       | input  | 必填，8-20 位，含字母和数字 |
| 学号       | input  | 必填，唯一，关联所属校区    |
| 姓名       | input  | 必填                        |
| 所属校区   | select | 下拉选择，数据来自校区表    |

**流程：**

1. 输入手机号 → 点击「获取验证码」→ 后端生成 6 位码写入 Redis（key: `sms:register:{phone}`，TTL 60s）
2. 填写完整信息 → 提交 → 后端校验学号唯一性 → 注册成功，自动登录，跳转首页
3. 注册成功奖励积分 **+50**

#### 3.1.2 登录

**页面：** `/login`

- 手机号 + 密码登录
- 登录成功后 Token 存入 `localStorage`，Pinia 维护用户状态
- 未登录用户访问需鉴权页面，Vue Router 守卫拦截并跳转登录页

#### 3.1.3 个人中心

**页面：** `/profile`

| 区块     | 内容                                                     |
| -------- | -------------------------------------------------------- |
| 基本信息 | 头像（MinIO 存储）、姓名、学号、学院、手机号（脱敏展示） |
| 数据统计 | 累计捐赠件数、累计领取件数、当前积分余额                 |
| 我的发布 | 跳转至个人发布列表，按状态 Tab 区分                      |
| 我的订单 | 跳转至个人订单列表，按状态 Tab 区分                      |
| 我的收藏 | 已收藏物品列表                                           |
| 积分明细 | 跳转积分明细页                                           |
| 修改密码 | 旧密码 + 新密码表单                                      |
| 退出登录 | 清除 Token，跳转登录页                                   |

### 3.2 物品广场

**页面：** `/` （首页）

#### 3.2.1 物品列表

- 采用卡片网格布局（手机端单列/双列，PC端多列）
- 默认按发布时间倒序，每页 12 条，滚动到底部自动加载（无限滚动）
- 每张卡片展示：封面图、物品名称/描述摘要、品类、尺码、新旧程度、获取方式（免费/积分数）

#### 3.2.2 搜索与筛选

| 筛选维度 | 组件     | 选项                         |
| -------- | -------- | ---------------------------- |
| 关键词   | 搜索框   | 模糊匹配名称、描述           |
| 品类     | 多选标签 | 上装、下装、外套、鞋帽、配饰 |
| 性别     | 单选     | 全部、男、女、通用           |
| 尺码     | 多选     | XS/S/M/L/XL/XXL/均码         |
| 新旧程度 | 多选     | 九成新、八成新、七成新及以下 |
| 获取方式 | 单选     | 全部、免费领取、积分兑换     |
| 所属校区 | 单选     | 下拉选择                     |

筛选条件以 URL query 参数形式保存，支持分享链接直达筛选结果。

#### 3.2.3 物品详情

**页面：** `/item/:id`

| 区块       | 内容                                  |
| ---------- | ------------------------------------- |
| 图片轮播   | 最多 6 张，支持点击放大               |
| 基本信息   | 品类、性别、尺码、新旧程度、发布时间  |
| 获取方式   | 免费领取 / 积分兑换（显示所需积分）   |
| 物品描述   | 发布者填写的详细描述                  |
| 发布者信息 | 头像、昵称（不展示真实姓名/学号）     |
| 操作按钮   | 收藏、申请认领（已认领/已完成时禁用） |
| 相似推荐   | 同品类其他物品，最多展示 6 条         |

### 3.3 物品发布模块

**页面：** `/publish`、`/publish/:id`（编辑）

#### 3.3.1 发布表单

| 字段     | 类型         | 规则                                       |
| -------- | ------------ | ------------------------------------------ |
| 物品图片 | 图片上传     | 1-6 张，单张 ≤ 5MB，上传至 MinIO，前端预览 |
| 物品名称 | input        | 必填，限 30 字                             |
| 品类     | select       | 必填：上装/下装/外套/鞋帽/配饰             |
| 性别     | radio        | 必填：男/女/通用                           |
| 尺码     | select       | 必填                                       |
| 新旧程度 | radio        | 必填：九成新/八成新/七成新及以下           |
| 物品描述 | textarea     | 必填，限 200 字                            |
| 获取方式 | radio        | 免费领取 / 积分兑换                        |
| 积分价格 | input number | 选择积分兑换时必填，范围 10-500            |
| 交付方式 | checkbox     | 支持勾选：线下自取、快递到门               |

#### 3.3.2 物品状态流转

```
[草稿] ──提交──> [待审核] ──通过──> [上架中] ──有人认领──> [交易中] ──确认完成──> [已完成]
                    │                   │
                 拒绝审核            主动下架
                    │                   │
                [已拒绝]            [已下架]
```

#### 3.3.3 发布规则

- 同一用户同时处于「上架中」+「交易中」状态的物品总数 ≤ 10 件
- 物品提交后进入待审核，后端审核通过才展示在广场
- 发布成功（审核通过后）奖励积分 **+30**

### 3.4 认领交易模块

#### 3.4.1 认领流程（免费领取）

```
学生点击「申请认领」
    → 填写备注（可选）
    → 生成订单（状态：待确认）
    → 通知发布方
    → 发布方在 24h 内确认 → 订单状态：待交接
    → 双方完成交接后，任一方点击「确认完成」
    → 订单完成，双方各获积分奖励
```

#### 3.4.2 认领流程（积分兑换）

```
学生点击「积分兑换」
    → 系统校验积分余额是否充足
    → 不足：提示去赚积分
    → 充足：冻结对应积分，生成订单（状态：待确认）
    → 后续同免费流程
    → 订单完成后：扣除冻结积分，积分转入发布方账户
    → 取消时：解冻积分，归还学生
```

#### 3.4.3 订单列表

**页面：** `/orders`

| Tab      | 说明               |
| -------- | ------------------ |
| 我发起的 | 以领取方身份的订单 |
| 我收到的 | 以发布方身份的订单 |

每条订单展示：物品封面、名称、订单状态、创建时间、操作按钮

#### 3.4.4 订单状态与操作

| 状态   | 发布方可操作 | 领取方可操作    |
| ------ | ------------ | --------------- |
| 待确认 | 同意 / 拒绝  | 取消申请        |
| 待交接 | 确认完成     | 确认完成 / 取消 |
| 已完成 | 查看详情     | 查看详情        |
| 已取消 | 查看详情     | 查看详情        |

#### 3.4.5 超时规则

| 场景         | 超时时长 | 处理                       |
| ------------ | -------- | -------------------------- |
| 发布方未响应 | 24 小时  | 订单自动取消，冻结积分解冻 |
| 待交接未完成 | 7 天     | 订单自动取消，冻结积分解冻 |

超时检测使用 Redis 延迟队列实现（`ZSET` 存储超时时间戳，定时 Job 扫描处理）。

### 3.5 积分模块

**页面：** `/points`

#### 3.5.1 积分规则

| 行为                           | 积分变动           |
| ------------------------------ | ------------------ |
| 注册成功                       | +50                |
| 成功捐赠（审核通过且订单完成） | +30                |
| 成功领取（订单完成）           | +10                |
| 完成订单评价                   | +5                 |
| 积分兑换衣物                   | -N（物品标定积分） |
| 积分兑换礼品                   | -N（礼品标定积分） |
| 管理员手动调整                 | ±N                 |
| 违规被处罚                     | -N（管理员设定）   |

#### 3.5.2 积分明细页

- 展示当前积分余额、冻结积分（交易中）
- 明细列表：来源说明、变动数量（红减绿增）、时间，支持分页

#### 3.5.3 积分礼品兑换

**页面：** `/points/mall`

- 展示管理员配置的礼品卡片（名称、图片、所需积分、库存）
- 点击「兑换」→ 确认弹窗 → 扣除积分 → 生成兑换码 → 线下凭码兑换（类似于取件码）

### 4.6 物品管理

**页面：** `/admin/items`

- 超级管理员：全平台物品
- 校区管理员：本校区物品

#### 4.6.1 物品列表

表格字段：封面图缩略图、物品名称、品类、发布者、所属校区、获取方式、状态、发布时间

支持筛选：状态（全部/待审核/上架中/交易中/已完成/已下架/已拒绝）、品类、校区、关键词

#### 4.6.2 审核功能

- 待审核列表优先展示（或单独 Tab）
- 点击「查看」进入审核详情页，展示物品完整信息（图片、描述等）
- 操作：**通过审核** / **拒绝审核**（拒绝需填写原因，通知发布者）
- 支持批量通过审核

#### 4.6.3 强制下架

- 对于上架中/交易中的物品，管理员可强制下架并填写原因
- 强制下架会同时取消该物品的进行中订单，冻结积分自动解冻
- 下架原因通知物品发布者

### 4.7 用户管理

**页面：** `/admin/users`

## 7. 签到模块

### 7.1 功能概述

用户每日签到可获得积分奖励，签到记录以 **GitHub Contribution 热力图** 风格按年展示，已签到的日期格子变为绿色，未签到为灰色，直观呈现全年参与活跃度。

------

### 7.2 Redis Bitmap 存储方案

#### Key 设计

```
sign:user:{userId}:{year}
```

示例：`sign:user:10086:2026`

#### 存储规则

- 一个 Bitmap 代表某用户某年的签到数据
- **offset** = 该天是当年第几天（从 0 开始，即 1 月 1 日 = offset 0）
- 签到时执行 `SETBIT sign:user:{userId}:{year} {dayOfYear} 1`
- 查询某天是否签到：`GETBIT sign:user:{userId}:{year} {dayOfYear}`
- 统计全年签到总天数：`BITCOUNT sign:user:{userId}:{year}`
- Key TTL：设置为 **2 年**，过期自动清理历史数据

#### 容量估算

- 每个 Key 最多 366 bit ≈ **46 字节**，极其轻量
- 1 万用户 × 2 年数据 ≈ **不足 1MB**

------

### 7.3 签到接口

#### 7.3.1 执行签到

```
POST /api/user/sign/today
```

**业务逻辑：**

1. 获取当前登录用户 ID
2. 计算今天是当年第几天（`dayOfYear - 1` 作为 offset）
3. `GETBIT` 检查今日是否已签到，若已签到返回提示
4. `SETBIT` 写入签到位
5. 计算连续签到天数（向前遍历 Bitmap，直到遇到 0）
6. 根据连续天数发放积分奖励，写入积分流水
7. 返回签到结果

**连续签到积分规则：**

| 连续签到天数 | 当日奖励积分 |
| ------------ | ------------ |
| 第 1 天      | +2           |
| 第 3 天      | +5           |
| 第 7 天      | +10          |
| 第 14 天     | +15          |
| 第 30 天     | +30          |
| 其他天数     | +2           |

#### 7.3.2 查询签到状态（当天）

```
GET /api/user/sign/today
```

返回：`{ signed: true/false, continuousDays: 7, todayPoints: 10 }`

#### 7.3.3 查询年度签到数据

```
GET /api/user/sign/year?year=2026
```

**业务逻辑：**

1. 使用 `BITFIELD sign:user:{userId}:{year} GET u{daysInYear} 0` 一次性读取全年 Bitmap
2. 解析每一位，生成日期 → 是否签到的映射数组
3. 返回给前端，格式如下：

```json
{
  "year": 2026,
  "totalDays": 366,
  "signedCount": 58,
  "continuousDays": 12,
  "data": [
    { "date": "2026-01-01", "signed": true },
    { "date": "2026-01-02", "signed": false },
    ...
  ]
}
```

## 8. 排行榜模块

### 8.1 功能概述

排行榜分为三个维度，帮助学生直观了解自身在校区中的活跃度排名，形成良性竞争氛围，激励更多参与。

| ------ | ------------------ | ------------------ |
| 积分榜 | 用户当前积分余额   | 实时（Redis ZSet） |
| 捐赠榜 | 成功捐赠的物品件数 | 实时（Redis ZSet） |
| 校区榜 | 各校区成交订单总数 | 每小时刷新         |

------

### 8.2 Redis ZSet 存储方案

#### Key 设计

```
rank:points:{campusId}        积分榜（按校区隔离）
rank:donate:{campusId}        捐赠榜
rank:campus:orders            校区榜（全平台）
```

#### 数据写入时机

| 事件                 | 操作                                                |
| -------------------- | --------------------------------------------------- |
| 积分变动             | `ZADD rank:points:{campusId} {newBalance} {userId}` |
| 捐赠完成（订单完成） | `ZINCRBY rank:donate:{campusId} 1 {userId}`         |
| 订单完成             | `ZINCRBY rank:campus:orders 1 {campusId}`           |

#### 查询

```java
// 取积分榜前 100 名（score 从高到低）
redisTemplate.opsForZSet().reverseRangeWithScores("rank:points:" + campusId, 0, 99);

// 查询当前用户排名（0-indexed，+1 得实际名次）
redisTemplate.opsForZSet().reverseRank("rank:points:" + campusId, userId);
```

### 8.5 数据初始化与一致性

#### 冷启动初始化

系统启动时，若 Redis 中对应榜单 Key 不存在，从数据库聚合查询并写入：

```sql
-- 积分榜初始化
SELECT user_id, point_balance FROM user WHERE campus_id = ? AND status = 0

-- 捐赠榜初始化
SELECT seller_id, COUNT(*) AS cnt FROM `order`
WHERE status = 2 AND campus_id = ? GROUP BY seller_id
```

#### 数据一致性保障

- Redis ZSet 为**主查询源**，数据库为持久化备份
- 积分变动、订单完成等核心事件，**先更新数据库，再更新 Redis**，保证不丢数据
- 每天凌晨 2:00 定时全量重算并覆盖写入 Redis ZSet（防止长期累积误差）
